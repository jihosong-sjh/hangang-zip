# Product Requirements

## 문서 목적
이 문서는 현재 `hangang.jihosong.com` 코드베이스를 기준으로, 서비스 보완과 확장을 위한 제품 요구사항을 정의한다.

## 현재 상태 요약
- 현재 서비스는 한강공원 11개를 지도에서 탐색하고 공원별 배달존과 주변 맛집을 보여주는 MVP다.
- 핵심 강점은 빠른 탐색 경험과 카카오맵 기반 장소 노출이다.
- 핵심 약점은 배달존 정확도와 운영 신뢰도다.
- 현재 배달존 데이터는 공식 확인 지점과 수동 후보 지점이 함께 섞여 있고, 운영 검수 체계는 문서/CSV 중심이다.
- 현재 서비스는 단일 화면 구조라 검색 유입, URL 공유, 재방문, 운영 관리 확장에 불리하다.
- 현재 백엔드는 `GET /api/parks`, `GET /api/parks/{id}`, `GET /api/delivery-zones/{zoneId}`까지 구현돼 있다.
- 공원 `slug` 컬럼은 DB에 존재하지만, 현재 public 상세 조회 계약은 `id` 기준이다.
- 배달존 공개 정책 필드명은 문서상의 `visibility`가 아니라 코드와 타입 기준 `displayPolicy`다.

## 제품 비전
`Hangang ZIP`을 단순 지도 MVP가 아니라, 한강 방문 전후 의사결정을 돕는 실행형 서비스로 확장한다.

핵심 비전:
- 사용자는 어느 한강공원에 갈지 빠르게 정할 수 있어야 한다.
- 배달존은 공식 여부와 신뢰도를 구분해 정확하게 안내해야 한다.
- 현장에서 실제로 찾기 쉬운 수령 지점과 주변 맛집을 함께 확인할 수 있어야 한다.
- 운영자는 후보 지점을 검수하고 공개 정책을 통제할 수 있어야 한다.

## 해결하려는 문제

### 1. 위치 신뢰도 문제
- 사용자는 “배달 가능한 공원”과 “실제 수령 가능한 정확한 지점”을 구분하기 어렵다.
- 공식 위치명은 있어도 좌표가 수동 보정값인 경우가 있다.
- 후보 지점과 공식 지점이 같은 UI 레벨로 보이면 신뢰가 떨어진다.

### 2. 탐색 정확도 문제
- 현재 맛집 검색은 공원 중심 좌표 또는 선택된 배달존 좌표에만 의존한다.
- 공원 중심 기준 검색은 실제 수령 동선과 다를 수 있다.
- 사용자는 기사에게 전달할 랜드마크나 접근 팁을 얻기 어렵다.

### 3. 성장 한계 문제
- 단일 지도 화면만으로는 검색 유입과 공유 확장이 어렵다.
- 사용자가 재방문할 이유가 약하다.
- “한강 배달존” 자체는 계절성과 빈도 한계가 있어 단독 주제로 큰 규모 확장이 어렵다.

### 4. 운영 한계 문제
- 현재 배달존 검수 메타데이터가 앱 모델보다 문서에 더 많이 남아 있다.
- 공개 정책, 검수 상태, 근거 링크, confidence score를 운영자가 관리하는 흐름이 없다.

## 대상 사용자

### 1차 사용자
- 한강공원에서 배달을 받아 피크닉, 모임, 야경, 데이트를 즐기려는 사용자
- 특정 공원을 아직 정하지 못했고, 목적에 맞는 공원을 비교하려는 사용자

### 2차 사용자
- 가족 나들이, 러닝, 산책, 야경 감상 등 한강 방문 의사결정을 하려는 사용자
- 공원별 분위기와 편의시설을 비교하려는 사용자

### 운영 사용자
- 배달존 후보를 검수하고 공개 정책을 관리하는 운영자
- 출처와 증거 링크를 확인하고 데이터 신뢰도를 조정하는 관리자

## 제품 목표

### Goal A. 정확한 배달존 안내
- 공식 배달존과 후보 지점을 명확히 구분한다.
- 사용자에게 랜드마크, 접근 팁, 주소/좌표, 출처, 확인일을 제공한다.
- 운영자가 공개 정책(`displayPolicy`)을 통제할 수 있다.

### Goal B. 한강 방문 의사결정 지원
- 공원 선택, 배달존 선택, 맛집 탐색, 방문 준비까지 하나의 흐름으로 연결한다.
- 활동 목적별 진입점을 만든다.

### Goal C. 검색 유입과 공유성 강화
- 공원별 상세 페이지, 컬렉션 페이지, 공유 가능한 URL을 제공한다.
- `한강 야경`, `가족 피크닉`, `배달 가능한 공원` 같은 의도형 유입을 노린다.

### Goal D. 운영 가능한 데이터 구조 확보
- 배달존을 독립 엔티티로 전환한다.
- 검수 증거, 공개 정책, confidence score, 만료 상태를 관리한다.

## 비목표
- 배달 주문 자체를 대행하거나 결제까지 처리하지 않는다.
- 모든 한강공원 배달존을 즉시 공식 수준으로 확정하지 않는다.
- 초기 단계에서 커뮤니티 기능을 과도하게 확장하지 않는다.
- 초기 단계에서 별도 모바일 앱 출시를 목표로 하지 않는다.

## 서비스 구조 방향

### `hangang.jihosong.com`
- 핵심 실행 서비스
- 공원 탐색, 배달존 상세, 주변 맛집, 동선/편의시설 안내

### `ops.jihosong.com`
- 내부 운영 도구
- 배달존 검수, 증거 링크 관리, 공개 정책 관리, confidence score 조정

### `today.jihosong.com`
- 상위 퍼널 서비스
- 오늘의 서울 나들이, 데이트, 산책, 야경, 피크닉 추천
- 사용자를 `hangang.jihosong.com` 상세 페이지로 연결

## 릴리스 범위

### Phase 1. 데이터 모델과 Public Read API 기반
- 배달존 독립 모델 도입
- evidence/review/access point supporting table 추가
- 기존 `GET /api/parks` 호환 유지
- `GET /api/delivery-zones/{zoneId}` 추가
- 공원 상세 식별자는 현재 `id` 기준으로 유지
- 상태:
  - 구현 완료
  - 단, `park_access_points`는 테이블만 준비됐고 public API/프론트에는 아직 연결되지 않음

### Phase 2. Hangang ZIP vNext Surface
- React Router 기반 `/parks/:parkSlug`, `/delivery-zones/:zoneId` 도입
- 단일 지도 화면을 유지한 공원 상세/배달존 상세 진입 구조 정리
- 배달존 기준 맛집 조회로 전환
- `displayPolicy` 기반 공개 정책 UI 정리
- 상태:
  - 미구현
  - 현재 Phase 2 진입 전 선행 정합성 정리 단계

### Phase 3. Ops Console
- 검수용 배달존 CRUD
- evidence/출처 관리
- 공개 정책과 confidence score 관리
- 로드뷰/지도 기준 검수 지원
- 상태:
  - 미구현

### Phase 4. Today Funnel
- 의도형 컬렉션 페이지
- 오늘의 추천 콘텐츠
- 공원 추천에서 실행 서비스로 연결
- 상태:
  - 미구현

## 핵심 지표

### 제품 품질
- 공개 배달존 중 공식 또는 고신뢰 지점 비율
- 배달존 상세 진입 후 맛집 클릭률
- 공원 상세 페이지 이탈률

### 성장
- 공원 상세 URL 유입 수
- 검색 유입 비중
- 공유 링크 재방문 비율

### 운영
- 검수 대기 지점 수
- evidence 부족 지점 수
- 마지막 확인일 초과 지점 수

## 주요 리스크
- 공식 데이터 소스가 제한적일 수 있다.
- 후보 지점을 너무 공격적으로 공개하면 신뢰도가 무너질 수 있다.
- `today.jihosong.com` 없이 `hangang.jihosong.com`만 키우면 검색 유입 확장이 느릴 수 있다.
- 운영 도구 없이 공개 서비스만 먼저 확장하면 데이터 부채가 커진다.

## 제품 원칙
- 저신뢰 데이터는 숨기거나 낮은 가시성으로 처리한다.
- 출처와 확인일이 없는 위치 정보는 운영 초안으로만 취급한다.
- 지도 UI보다 데이터 신뢰도와 사용 맥락을 우선한다.
- URL 공유와 검색 유입이 가능한 구조를 기본 전제로 설계한다.
- 문서의 목표 용어보다 현재 코드 계약을 우선하며, 현재 공개 정책 필드명은 `displayPolicy`로 맞춘다.
